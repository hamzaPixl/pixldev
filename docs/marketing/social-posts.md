## 4. Fine-Tuned Local Models Are the Next Layer

**Visual:** /illustrations/blog/fine-tuned-local-models-are-the-next-layer.jpg

### LinkedIn (EN)

Prompting is the fastest way to reach early usefulness. Then the bill arrives.

Too much behavior lives in the prompt. Instructions grow. Edge cases pile up. You're paying a frontier model to re-read the same instructions forever. It never gets bored. Your budget does.

So the next layer, for me, is specialized fine-tuned local models. Not smarter. Compressed.

→ Frontier models for broad reasoning and hard synthesis.
→ Structured systems for memory, retrieval, gating.
→ Small fine-tuned models for narrow, repeated jobs.

Fine-tuning compresses repeated behavior into the model, so the system needs less scaffolding at runtime. Less prompt, less latency, less variance.

The honest caveat: this isn't reasoning replaced by brittle automation. Reasoning stays central. Specialized models just take the low-entropy work that should already be stable.

The next step isn't more prompting. It's better specialization.

Full argument in the first comment.

### LinkedIn (FR)

Le prompting, c'est le chemin le plus rapide vers une première utilité. Puis la facture arrive.

Trop de comportement vit dans le prompt. Les instructions grossissent. Les cas limites s'empilent. Tu paies un modèle frontier pour relire les mêmes instructions à l'infini. Lui ne se lasse jamais. Ton budget, si.

Alors la prochaine couche, pour moi, c'est des modèles locaux fine-tunés et spécialisés. Pas plus intelligents. Compressés.

→ Les modèles frontier pour le raisonnement large et la synthèse difficile.
→ Les systèmes structurés pour la mémoire, la retrieval, le gating.

Le fine-tuning compresse le comportement répété dans le modèle, pour que le système ait besoin de moins de scaffolding au runtime. Moins de prompt, moins de latence, moins de variance.

L'aveu honnête : ce n'est pas le raisonnement remplacé par de l'automatisation fragile. Le raisonnement reste central. Les modèles spécialisés prennent juste le travail à faible entropie qui devrait déjà être stable.

La prochaine étape, ce n'est pas plus de prompting. C'est une meilleure spécialisation.

## 6. No Consumer, No Build

**Visual:** /illustrations/blog/no-consumer-no-build.jpg

### LinkedIn (EN)

I pointed my own tooling at my own fleet and ran a code-verified audit. Every repo, every service, every deployment.

The most expensive thing it found wasn't a bug. It was working software with zero callers.

→ A live company-data service, four real sources integrated. Zero callers.
→ Zero cross-service calls in production. The platform was a diagram.
→ Brand identity re-implemented four times.
→ A gateway with a budget guard that hardcoded cost to zero, so it shipped without ever charging anyone.

None of it is broken. It all passes its tests. That's what makes it expensive. It's inventory dressed as assets.

So the doctrine has one load-bearing rule: no consumer, no build. Infrastructure gets built the moment a real product pulls it into existence, and not one sprint earlier.

The honest caveat: this rule has a price, and I'm paying it. Sometimes that means deleting working software instead of rescuing it.

Code with no callers isn't an asset. Link in the first comment.

## 7. Robotics Is Where Agentic Systems Become Real

**Visual:** /illustrations/blog/robotics-is-where-agentic-systems-become-real.jpg

### LinkedIn (EN)

My first posts were about the boring layers: execution, memory, fine-tuning, reliability. For me they all point one way. Robotics.

I don't see agents as the destination. I see them as the road to systems that act in the physical world. Agents matter because they force the hard problems first:

→ planning
→ memory and reasoning
→ specialization and evaluation
→ reliability and adaptation

Those are exactly the layers a robot needs before it earns its place. A robot without them is mostly hardware.

And shape isn't the point. Most useful robots aren't humanoids. Drones, arms, wheeled machines, small classroom companions. The market won't wait for a perfect humanoid to start changing work.

The honest near term: none of this replaces people. The job is to make humans more capable again. Direction stays human.

First the system learns to reason. Then to act. Then it enters the room. Link in the first comment.

## 11. The Baton Pattern

**Visual:** /illustrations/blog/the-baton-pattern.jpg

### LinkedIn (EN)

Put three agents in a pipeline. One plans, one builds, one reviews. Each needs to know what the others decided. Not the full transcript. The gist.

Two ways this goes wrong: pass the full artifacts and you burn tokens on context nobody reads twice. Pass nothing structured and the next agent misses the why, then contradicts a decision made two stages back.

So the fix is small. A JSON object, around 1,000 tokens, that travels with the work.

→ Each agent reads it before starting.
→ Patches only the fields that changed when done.
→ decision_log is append-only, so nobody quietly reverses a call.
→ Every patch is stamped, so you can read exactly where the context bent.

It's a relay baton that took notes.

The honest limit: the baton carries context, not proof. It tells the next agent what happened and why. It does not tell you the work was any good.

One less way for a pipeline to lie to itself. Link in the first comment.

## 13. Under the Hood: Harness + Knowledge OS

**Visual:** /illustrations/blog/the-technical-stack-behind-my-ai-projects.jpg

### LinkedIn (EN)

Almost everything I ship sits on two layers, and I keep them apart on purpose.

→ A harness that drives execution.
→ A Knowledge OS that supplies context.

I don't want planning, memory, retrieval, and judgment collapsed into one long prompt. Each layer gets one job.

The harness is a loop, not a single generation step. Plan into a real spec. Generate against it. Evaluate the actual result. Gate: stop, iterate, or escalate. The point is the split between building and judging. The layer that produces is not the layer that decides it's good enough. That gap is most of the distance between a demo agent and something you can run twice.

The Knowledge OS treats knowledge as structure, not search. Ingestion, entity extraction, relation mapping, conflict detection, versioning.

The honest caveat: I'll describe the shape, the roles, the reasoning. I won't publish the prompts and heuristics closest to production leverage. That's the line between sharing the design and handing over the machine.

The harness helps the system do. The Knowledge OS helps it know. Link in the first comment.
